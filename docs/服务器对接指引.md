# 抽卡平台 → 游戏服务器 对接指引

> 这是一份面向非专业开发人员的说明文档，解释抽卡平台是如何连接游戏服务器、调用抽卡功能的。

---

## 一、整体结构简述

抽卡平台是一个**网页工具**（浏览器打开），后面跑着一个 **Python 服务**。它的工作流程可以理解为：

```
你的浏览器 ──HTTP网页请求──▶ Python 后台服务(Flask) ──TCP网络连接──▶ 游戏服务器(C#)
```

简单来说：
- **浏览器**：负责显示界面、点按钮
- **Python 后台**：负责"翻译"——把浏览器的操作翻译成游戏服务器能听懂的格式发过去，再把服务器的回复翻译回来给浏览器显示
- **游戏服务器**：真正执行抽卡逻辑的地方

---

## 二、连接游戏服务器的完整过程

### 第1步：找到"真正的服务器地址"

游戏服务器有一个"前台（RouterManager）"和一个"后台（XServer）"：
- **RouterManager**：相当于前台接待，告诉你"真正干活的服务器在哪"
- **XServer**：真正处理抽卡请求的服务器

所以连接时先问 RouterManager：

```
抽卡平台 ──HTTP请求──▶ RouterManager(比如 10.20.200.20:40500)
                         "请问 XServer 在哪？"
    
RouterManager 回复：    "XServer 在 10.20.200.20:2404"
```

> 对应代码：`game_rpc_client.py` 里的 `resolve_xserver()` 方法

### 第2步：建立网络长连接

拿到 XServer 的真实地址后，Python 后台通过 **TCP 长连接** 连上去（就像打电话一样，建立一条持续的通话线路）。

> 对应代码：`game_rpc_client.py` 里的 `connect()` 方法

### 第3步：登录

连上之后，发送登录请求（账号+密码）。服务器验证通过后会回一条"登录成功"的消息。

```
抽卡平台 ──发送──▶ Login(账号="kk", 密码="1234")
游戏服务器 ──回复──▶ LoginResult { result: true, msg: "登录成功" }
```

> 对应代码：`game_rpc_client.py` 里的 `login()` 方法
> 对应路由：`game_server_routes.py` 里的 `auto_connect()` 把上面3步串起来一键完成

## 三、通信协议（数据怎么传）

### 消息格式

每条消息都长这样：

```
[4个字节标记消息长度] + [JSON 文本内容]
```

JSON 内容叫做 **BasicMsg**（基础消息），有3个字段：

| 字段 | 含义 | 例子 |
|------|------|------|
| rpcId | 要调用的方法名 | "GetCardPoolList"、"GachaCardPool" |
| returnId | 消息编号（用来匹配请求和回复） | 1, 2, 3… |
| msg | 具体参数（经过base64编码） | "eyJwb29sSWQiOjF9" |

**为什么 msg 要 base64 编码？**
因为游戏服务器用的是 C# 语言，C# 把 byte[]（字节数组）转成 JSON 时默认用 base64 编码。所以 Python 这边发送和接收 msg 字段时都要做 base64 的编解码。

> 对应代码：`game_rpc_client.py` 里的 `_send_rpc()` 方法（发送时编码）和 `_on_message()` 方法（接收时解码）

## 四、调用了哪些游戏服务器接口

抽卡平台目前用到了以下游戏服务器接口：

### 4.1 获取卡池列表

| 项目 | 内容 |
|------|------|
| 接口名 | `GetCardPoolList` |
| 参数 | 无 |
| 返回 | 卡池列表，每个卡池包含：poolId(编号)、times(已抽次数)、isNew(是否有新标记)、ssrNeedTimes(保底还差几抽) |
| 对应C#接口 | `IPlayerGachaServer.GetCardPoolList()` |
| 对应平台路由 | `GET /api/game/pools` |

### 4.2 抽卡

| 项目 | 内容 |
|------|------|
| 接口名 | `GachaCardPool` |
| 参数 | poolId(卡池编号)、times(0=单抽, 1=十连) |
| 返回 | ErrorCode(错误码) + GachaResponse(抽卡结果)，结果包含：抽到的卡牌列表(cardId, isNew)、更新后的卡池信息 |
| 对应C#接口 | `IPlayerGachaServer.GachaCardPool(int poolId, EnumGachaTimes times)` |
| 对应平台路由 | `POST /api/game/pull/single`（单抽）、`POST /api/game/pull/multi`（十连） |

**常见错误码：**
| 错误码 | 含义 |
|--------|------|
| 0 | 成功 |
| 24 | 抽卡货币不足 |
| 31 | 背包空间不足 |
| 33 | 钻石不足 |
| 38 | 卡池已关闭 |

### 4.3 清除卡池新标记

| 项目 | 内容 |
|------|------|
| 接口名 | `ReadCardPoolNew` |
| 参数 | poolId(卡池编号) |
| 返回 | 无 |
| 对应C#接口 | `IPlayerGachaServer.ReadCardPoolNew(int poolId)` |
| 对应平台路由 | `POST /api/game/pools/{poolId}/read` |

### 4.4 登录

| 项目 | 内容 |
|------|------|
| 接口名 | `Login` |
| 参数 | name(账号)、password(密码) |
| 返回 | 无直接返回，服务器通过推送 `LoginResult` 通知结果 |
| 对应C#接口 | `ILoginServer.Login(string name, string password)` |

## 五、卡牌数据从哪来

服务器返回的抽卡结果只有 **cardId**（卡牌编号），没有卡牌名字和品质。

为了让前端显示名称和稀有度，Python 后台会读取项目里的一个配置文件：

```
Server/Config/Json/cs/CardMainSet.json
```

这个文件里存了所有卡牌的信息（编号、名字、品质等）。Python 后台读取后做一个映射：
- quality = 0 → SSR
- quality = 1 → SR
- quality = 2 → R

> 对应代码：`game_server_routes.py` 里的 `_load_card_data()` 和 `_enrich_card()`

## 六、数据流转全链路（以"抽卡十连"为例）

```
1. 你点击"抽卡10次"按钮
      ↓
2. 浏览器发 HTTP POST 到 /api/game/pull/multi，带 { pool_id: 1 }
      ↓
3. Python 后台收到请求，翻译成 BasicMsg：
   {
     rpcId: "GachaCardPool",
     returnId: 5,
     msg: base64编码后的 {"poolId": 1, "times": 1}   ← times:1 表示十连
   }
      ↓
4. 通过 TCP 发给游戏服务器
      ↓
5. 游戏服务器执行抽卡逻辑，返回结果（也是 BasicMsg）
      ↓
6. Python 后台收到结果，解码 base64，解析出：
   - ErrorCode: { Code: 0, isSuc: true }   ← 成功
   - GachaResponse: {
       pool: { poolId: 1, times: 11, ... },   ← 已抽次数更新了
       cards: [{ cardId: 1001, isNew: true }, { cardId: 2003, ... }, ...]
     }
      ↓
7. Python 后台用 CardMainSet.json 补充每张卡的名字和稀有度
      ↓
8. 返回给浏览器：
   {
     success: true,
     pool: { poolId: 1, times: 11, ... },
     cards: [
       { cardId: 1001, name: "某SSR角色", rarity: "SSR", isNew: true },
       { cardId: 2003, name: "某R角色", rarity: "R", isNew: false },
       ...
     ]
   }
      ↓
9. 浏览器收到数据，显示抽卡结果动画
```

## 七、如果服务端接口有修改，如何跟进

### 关注哪些文件

当服务端同事说"抽卡接口改了"，你需要关注以下 C# 文件（它们是"接口定义"，定义了发什么、收什么）：

| 要关注的 C# 文件 | 影响范围 |
|-----------------|---------|
| `Shared/Protocols/PlayerActor/PlayerGachaRpc.cs` | 抽卡相关所有接口和数据结构 |
| `Shared/Protocols/LoginActorRpc.cs` | 登录相关接口 |
| `Shared/Protocols/PlayerActor/PlayerRoleRpc.cs` | Ping/获取角色数据 |
| `Shared/XUtility/ErrorUtility.cs` | 新增错误码时 |
| `Shared/XActor/DataSender.cs` | 消息信封格式变更（极少改） |

### 修改对照表

当 C# 那边改了接口，你需要在 Python 这边做对应修改：

| C# 那边改了什么 | Python 这边怎么改 | 改哪个文件 |
|----------------|------------------|-----------|
| 接口名变了（比如 `GachaCardPool` 改名） | 修改 `_send_rpc("GachaCardPool", ...)` 里的方法名 | `game_rpc_client.py` |
| 接口参数变了（比如多加一个参数） | 修改对应方法里传给 `_send_rpc` 的参数字典 | `game_rpc_client.py` |
| 返回值结构变了（比如 GachaResponse 多了一个字段） | 修改解析逻辑 | `game_rpc_client.py`（解析方法）+ `game_server_routes.py`（HTTP返回） |
| 数据类变了（比如 GachaPool 多了字段） | 修改对应的 Python dataclass 和解析/返回逻辑 | `game_rpc_client.py` + `game_server_routes.py` |
| 新增错误码 | 前端 JS 里的错误码映射加一条 | `gacha-engine.js` 里的 `_errorCodeMap` |
| 新增接口（比如新的抽卡方式） | Python 端加新方法，Flask 加新路由，前端加对应调用 | 都要改 |
| CardMainSet.json 表结构变了 | 修改 `_load_card_data()` 的字段读取 | `game_server_routes.py` |

### 具体修改步骤（举例：假设服务端给 GachaCardPool 加了一个新参数 `useTicket`）

**第1步**：确认 C# 接口变成了什么
```csharp
// 改之前
GachaCardPool(int poolId, EnumGachaTimes times)
// 改之后 — 多了 useTicket 参数
GachaCardPool(int poolId, EnumGachaTimes times, bool useTicket)
```

**第2步**：修改 `game_rpc_client.py` 中对应的方法
```python
# 改之前
def gacha_card_pool(self, pool_id, times):
    result = self._send_rpc("GachaCardPool", {
        "poolId": pool_id,
        "times": int(times)
    })

# 改之后 — 加上 useTicket
def gacha_card_pool(self, pool_id, times, use_ticket=False):
    result = self._send_rpc("GachaCardPool", {
        "poolId": pool_id,
        "times": int(times),
        "useTicket": use_ticket       # ← 新加的
    })
```

**第3步**：修改 `game_server_routes.py` 中的路由（如果前端也需要传这个参数）
```python
# 在 pull_single() 和 pull_multi() 里加上
use_ticket = data.get('use_ticket', False)
error_code, response = client.gacha_card_pool(pool_id, EnumGachaTimes.One, use_ticket)
```

**第4步**：如果前端需要让用户控制这个选项，修改前端 JS

## 八、平台服务器配置说明

### 配置文件位置

`card_platform/config.py`

### 服务器连接配置

```python
GAME_SERVER_CONFIG = {
    'host': '10.20.200.20',    # RouterManager 地址
    'port': 40500,              # RouterManager 端口（不是最终连接的端口！）
    'timeout': 10,              # 请求超时秒数
    'username': 'kk',           # 默认登录账号
    'password': '1234',         # 默认登录密码
}
```

### 可选服务器列表

```python
GAME_SERVER_LIST = [
    {'id': 'dev', 'name': 'QA服', 'host': '10.20.200.20', 'port': 40500},
    {'id': 'dev', 'name': '英雄服', 'host': '10.20.200.20', 'port': 40300},
    {'id': 'dev', 'name': '内网服', 'host': '10.20.200.20', 'port': 40100},
]
```

要加新服务器，在这个列表里加一项即可，前端侧边栏会自动显示。

## 十、常见问题

### Q: 连接不上服务器怎么办？
检查 `config.py` 里的 `GAME_SERVER_CONFIG` 的 host 和 port 是否正确。这里填的是 **RouterManager** 的地址和端口，不是 XServer 的。

### Q: 报错 "Invalid frame length: 1347703880" 是什么？
这个数字对应 ASCII "POST"，说明连错了——你把 HTTP 端口当 TCP 端口连了。需要通过 RouterManager 解析出真正的 XServer TCP 端口。

### Q: 报错 "玩家已在线" 怎么办？
说明上次连接没正常断开，服务端还认为该玩家在线。平台会自动重试（断开 → 等待 → 重连）。如果还是不行，等1-2分钟让服务端超时清理。

### Q: 抽卡报错码 24 是什么？
货币不足。需要通过 GM 命令给测试账号加抽卡货币。

### Q: 卡池突然消失了怎么办？
`GetCardPoolList` 只返回服务端**当时开放**的卡池。如果卡池有时间限制或被服务端关闭了，就不会出现在列表里。找服务端同事确认卡池配置。
